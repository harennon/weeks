Native Share… — the fix, made visible

The share sheet from src/index.html, showing how the pre-rendered file fix changes the button lifecycle. Use the state chips under each option to preview idle / preparing / failed.

Why it broke, and what the UI has to express

navigator.share() needs transient user activation — it must run synchronously inside the click. Today it runs inside the async toBlob() callback, so the gesture has expired and the browser rejects with NotAllowedError. The fix pre-renders the PNG File ahead of the tap, so the click calls share() directly. The only new UI concept is a "file readiness" state on the Share button.

Before — broken click → toBlob() (async) → share() fires on a later task → activation gone → generic "Couldn't share" toast.
After — fixed sheet opens / format toggled → render File & cache it → click reads cached file → share() fires synchronously → OS sheet opens.
Option A

Explicit readiness

The Share button reflects file state: it shows a brief Preparing… spinner while the cached PNG renders, then becomes tappable. Honest about the async work; guarantees the tap only fires when a file exists.

State

Idle / ready: the cached File exists, so a tap calls share() synchronously. This is the steady state — the OS sheet opens instantly.

Option B

Invisible prep, silent recovery

The Share button never changes appearance. Prep happens silently in the background; a tap always uses the freshest cached file. Failure is rare — when it happens, a quiet inline hint points at Download instead of a disappearing toast.

State

Idle / ready: button looks identical at all times. The cached file is refreshed on open and on every format toggle; the tap reads whatever is cached. Lowest chrome.

Both options share the same underlying fix (pre-rendered cached File, synchronous share() in the click). They differ only in how much of the "file readiness" state the UI exposes.

Recommendation: Option A — the spinner keeps the tap honest and covers the cold-font-load repaint window.
